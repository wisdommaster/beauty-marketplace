# Детальная механика — недостающие блоки

## 1. No-show и рейтинг пользователя

### Правило

```
3 инцидента за 30 скользящих дней → ограничение
```

### Инциденты

| Тип | Засчитывается | Описание |
|---|---|---|
| NO_SHOW | Да | Бронь была CONFIRMED/PAID, пользователь не пришёл. Выявлено через 2ч после окончания слота (нет статуса COMPLETED) |
| LATE_CANCEL | Да | Пользователь отменил ≤30 мин до начала (OFFLINE) или ≤30 мин до начала без refund (ONLINE) |

### Не засчитывается

| Ситуация | Причина |
|---|---|
| Отмена >30 мин до начала | Штатная, не инцидент |
| Салон отменил бронь | Не вина пользователя |
| Сбой YClients, двойная бронь | Не вина пользователя |

### Шкала ограничений

| Инцидентов за 30 дней | Ограничение |
|---|---|
| 3 | Только ONLINE_REQUIRED (предоплата) на 7 дней |
| 5 | Блокировка бронирований на 14 дней |
| 7+ | Блокировка аккаунта, ручная модерация |

### Счётчик в профиле

```
Надёжность бронирований: ████████░░ 80% (4 из 5)
Последние 30 дней: 0 инцидентов
```

## 2. Возвраты (ONLINE_REQUIRED)

### Условия

| Сценарий | Возврат | Инициатор |
|---|---|---|
| Отмена >30 мин до начала | 100% | Пользователь |
| Отмена ≤30 мин до начала | 0% | Пользователь |
| Отмена салоном в любое время | 100% | Бизнес / платформа |
| Двойная бронь (ошибка платформы) | 100% + извинения | Платформа |
| Слот снят с публикации до брони | Нет оплаты → нет возврата | — |

### Процесс возврата

```
1. Пользователь → «Отменить» в /app/bookings
2. Платформа проверяет: время до начала
3. >30 мин → YooKassa: refund(payment_id, amount=100%)
4. YooKassa → webhook → платформа: статус CANCELLED, refund_status=completed
5. YClients: отмена брони (DELETE/PATCH)
6. Слот: booked_count -= 1, возвращается в пул
```

### Сроки

- YooKassa refund: до 5 рабочих дней (зависит от банка)
- Платформа показывает статус refund: pending / completed

## 4. Рейтинг студий

### Источник: Яндекс.Карты

Рейтинг загружается из API Яндекс.Карт для каждой студии при регистрации и обновляется раз в сутки.

- Шкала: 1-5 звёзд
- Обновление: ежедневный поллинг
- Если студии нет на Яндекс.Картах: рейтинг не показывается. Студия может добавить организацию в Яндекс.Бизнес.

Отображение: ⭐ 4.8 (Яндекс), кол-во оценок.

### Отказ от внутренней системы рейтинга

- Нет сбора отзывов внутри платформы
- Нет защиты от накрутки
- Нет проблемы холодного старта (0 отзывов)
- Яндекс.Карты — доверенный внешний источник

### Резервный источник (если нет в Яндекс)

- 2ГИС API
- Google Maps (для зарубежных студий в будущем)
- Если нет нигде → «Недостаточно оценок»

## 5. Антифрод

### GPS-спуфинг

| Метод | Проверка |
|---|---|
| GPS координаты | Основной источник |
| IP-геолокация | Сравнение с GPS (дельта >5 км = подозрение) |
| WiFi-сети вокруг | Если доступен WiFi, сравнение BSSID с геобазой |

При расхождении GPS/IP >5 км: запрос подтверждения через фотографию окружения (один раз).

### Мульти-аккаунты

- Один телефон = один аккаунт
- Проверка: device fingerprint + телефон
- Новый аккаунт с того же устройства: флаг, ручная проверка при 3+ аккаунтах

### Фейк-брони (бизнес бронирует сам себя для рейтинга)

- Проверка: пользователь и бизнес с одного устройства / IP
- Проверка: аккаунт пользователя создан <24ч назад и сразу бронь
- Паттерн: множественные брони → отмены от одного пользователя в одной студии

## 6. Модерация бизнесов

### При регистрации

| Проверка | Автоматически |
|---|---|
| Валидность YClients API-ключа | Да (проверка соединения) |
| Категория соответствует адресу | Да (map API проверка) |
| Название не содержит запрещённых слов | Да (фильтр) |

Ручная модерация: выборочная, 10% новых бизнесов, фокус на премиум-категории.

### Блокировка

| Причина | Действие |
|---|---|
| 3 жалобы от пользователей | Временная блокировка, запрос объяснений |
| Систематические отмены броней бизнесом | Понижение рейтинга надёжности |
| Фейк-брони (доказано) | Перманентная блокировка |
| Неактивность 30 дней | Автоматическая приостановка (слоты скрыты) |

## 7. Хранение YClients API-ключа

### Модель

- Ключ шифруется на уровне БД (AES-256)
- Ключ никогда не показывается целиком в интерфейсе: `yc_sk_****a1b2`
- При смене ключа бизнесом: старый инвалидируется, новый проверяется → сохраняется
- Логгирование: факт использования ключа (успех/ошибка), не сам ключ

### Ротация

- Платформа не инициирует ротацию
- Если YClients возвращает 401: уведомление бизнесу «Ключ API недействителен. Обновите.»
- Слоты не снимаются мгновенно — grace period 1 час, затем скрываются

## 8. Мульти-локация (Maximum)

### Правила

- Один бизнес-аккаунт → до 5 адресов
- Каждый адрес: свой YClients API-ключ (или единый для сети)
- Слоты публикуются раздельно по адресам
- Лимит слотов/нед: общий на все адреса (21)
- Карточка в ленте: адрес конкретной локации, остальное — общее (название сети)

### Интерфейс

- `/business/locations` — список адресов
- «Добавить адрес»: поле адреса, YClients-ключ для этой локации (если отдельный)
- При публикации слота: выбор адреса из выпадающего списка

## 9. Free-тариф (Start) — механика окончания

### Триал

- 14 дней с момента регистрации
- Счётчик в `/business/tariff`: «Осталось 5 дней»
- За 3 дня до конца: уведомление «Выберите тариф»

### По окончании

```
День 15:
- Все опубликованные слоты скрываются из ленты
- Бизнес не может публиковать новые
- Данные сохраняются (услуги, настройки)
- Бизнес видит заглушку: «Тариф истёк. Выберите план.»

День 30 без тарифа:
- Данные могут быть удалены (GDPR)
```

## 10. Конфликт бронирования — детальный флоу

### Сценарий: два пользователя жмут «Забронировать» на один слот

```
t=0.00  A жмёт «Забронировать» (последнее место: 1/4 → 2/4)
t=0.05  B жмёт «Забронировать» (последнее место: 1/4 → видит «1 место»)

t=0.10  Платформа обрабатывает A:
         BEGIN TRAN
           SELECT slot WHERE id=X FOR UPDATE
           booked_count=1, capacity=4 → 1 < 4 → OK
           UPDATE booked_count=2
           INSERT booking (user=A, status=HELD)
         COMMIT

t=0.15  Платформа обрабатывает B:
         BEGIN TRAN
           SELECT slot WHERE id=X FOR UPDATE
           booked_count=2, capacity=4 → 2 < 4 → OK
           UPDATE booked_count=3
           INSERT booking (user=B, status=HELD)
         COMMIT

Оба успешны — но в параллели, атомарно, без гонки.
```

### Сценарий: последнее место, два пользователя

```
t=0.00  A жмёт «Забронировать» (3/4)
t=0.05  B жмёт «Забронировать» (3/4)

t=0.10  A: FOR UPDATE, 3<4 → OK, booked_count=4
t=0.15  B: FOR UPDATE, 4<4 → FALSE → ROLLBACK

B получает: { error: "slot_taken", alternatives: [...] }
```

## 11. Композиция ленты — формула scoring

```
Для каждого слота в пуле:

distance_factor = 1 - (distance / user_radius)          // 0..1, ближе = выше
discount_factor = discount / max_possible_discount       // 0..1, больше скидка = выше  
urgency_factor  = 1 - (minutes_until_start / 720)       // 0..1, 720 мин = 12ч
                                                        // скорее = выше

base_score = (distance_factor × 0.35) + (discount_factor × 0.40) + (urgency_factor × 0.25)

Применяем модификаторы:
- preferred_category:  score × 1.5
- смежная_категория:    score × 1.0
- discovery:             score × 0.8
- бизнес_приоритет:      score × tariff_priority (1.0 / 1.5 / 2.5)

Сортируем по score DESC.
Берём top-10.
```

## 15. Yandex Maps API — интеграция

### Данные для карточки

- Рейтинг: `GET /maps/api/place/{place_id}` → rating, reviews_count
- Фото: `GET /maps/api/place/{place_id}/photos` → url (до 5 фото)
- Адрес → координаты: Geocoder API
- Маршруты: `yandexmaps://` deeplink для 3 пинов

### Частота

- Рейтинг: раз в 24 часа
- Фото: раз в 7 дней
- Маршруты: runtime, на каждый запрос
